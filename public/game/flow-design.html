Implementation notes / Risks & open questions

  • Role assignmentReveal roles client-side at the instant dispersal hits 0:00. Server writes criminalUids once when the dispersal timer expires; clients listen for that field flipping null → array and trigger the reveal animation.
  • Runner lock clockEach Runner gets an independent 5-min timer starting from their last successful position-photo upload. Position-drift threshold (~30 m, TBD) and violation handling (warn vs. auto-loss) still need a decision.
  • Stalemate detection"15 min without a capture" — the timer resets on every capture event. The first stalemate after the first capture starts counting from that capture, not from game start.
  • Cancel = full abortAny cancel during Pre-Game Ready returns everyone to Lobby after 3s. The cancel must sync via Firestore (not local-only).
  • Host-only startOnly Host sees the primary CTA in Lobby. After their photo, other players see a "waiting for Host to start" placeholder.
  • No bottom dock in sub-screensAll sub-screens hide the dock; only the four main tabs (Start / Ranks / Social / Me) show it. Toggle via body[data-game-active] or a separate layout shell.
  • Chat scopePublic chat is on in Lobby + Dispersal. Whether it stays during Hunt is undecided — leaning toward keeping it as a collapsed bottom-right widget so people can taunt without leaking location.
  • Position photo: people in frame?This design assumes "position photo" = environment shot (street corner, storefront), not a selfie. Confirm whether the camera should invert face-detection (reject faces).
  • Duration rangesDispersal 1–5 min, Hunt 20–60 min, Lock 3–10 min, Stalemate 10–20 min. Steppers in the create form (already shown in this design).
  • QR capture preservedThe existing QR mutual-scan capture flow is kept — entry point in the Hunter view's top-right scan button. New rule needed: should Runners only be scannable while unlocked? TBD.
